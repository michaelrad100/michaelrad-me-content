Over the last couple of months I've connected a lot of tools to my AI setup. Shopify, Stripe, Klaviyo, Circle, Amazon, Google Analytics, QuickBooks. In every case, I had the option of going the MCP or API route.

MCP is getting a lot of hype right now, and for good reasons. You can connect a service in seconds with no code, the AI can use it live in the conversation, and new connectors are showing up for just about everything. It's all over my feeds. And yet, more often than not, I ended up walking through the plain old API door instead.

Every time, I'd start with the MCP because it was the quick, shiny option. Partway in I'd hit a wall, or just notice the API route gave me more of the service to work with and more room to change things later. So I'd switch.

As someone still understanding the nuance of working with both, here's what I learned:

**MCP** ([Model Context Protocol](https://modelcontextprotocol.io/)) plugs a service straight into the AI chat, so the model can use it live while we're talking. Think of it as handing the assistant a remote control for an app. Flip it on, no code, and it just works in the conversation. 

**An API (Application Programming Interface) connection** is a little script that goes and fetches the data on command. It gets written once. After that it runs on its own, no person and no AI needed.

**What actually separates them.**

> You can schedule either one. The difference is that a scheduled MCP still has to wake the AI up to run. A script is just code, it runs on its own.

Both can be automated, but what happens at run time is completely different. My API scripts run on a scheduler on my Mac with no AI in the loop at all. The AI shows up once, to write the script, and after that it's plain code firing in the background (I could even move it onto an always-on server so my laptop could be shut). An MCP runs every request back through the AI, so it costs tokens every single time, and just being connected loads its instructions into the model's memory before I've done anything. For a report I'll run every month for years, that gap adds up.

The token savings were just a bonus. The real driver was that the API almost always gave me access to more of the source's data than the MCP did.

Take the monthly report I keep. It pulls key numbers from Shopify, Stripe, Amazon (which has no MCP at all), Google Analytics, Search Console, and QuickBooks into one Google Sheet, every month, without me babysitting it. Done through MCPs, that job would mean running an AI session every month, hoping each connection behaves, paying tokens for every pull, and settling for whatever slice of each tool's data the connector exposes. As scripts it just happens on a schedule, the same way every time, for free, pulling exactly the fields I asked for.

Here's how the rest shook out, tool by tool:

- **Circle** I started on MCP, and it kept breaking, a string of errors that finally ended in it locking me out entirely. The API has been quiet ever since.
- **Klaviyo** stayed on MCP. It's great when I want to poke at email campaigns mid-conversation and ask follow-up questions.
- **Slack, Gmail, Drive, and Calendar** are all MCP too, and they're a joy, because I'm always right there in the chat when I use them.

That last group is exactly what MCP is built for.

So the rule I landed on is less about which is better and more about what the job needs.

|                               | MCP (the cockpit)                                 | API connection (the autopilot)                         |
| ----------------------------- | ------------------------------------------------- | ------------------------------------------------------ |
| **Best for**                  | Sitting with the AI, exploring, asking follow-ups | A job that has to run on a schedule, alone             |
| **Setup**                     | Flip it on, no code                               | Write a small script once (minutes for the AI to code) |
| **Needs Claude open to run?** | Yes, it works only inside a live chat             | No, it runs on its own in the background               |
| **Token cost**                | Spends tokens every time it's used                | Free to run once it's built                            |
| **When it breaks**            | You wait on whoever built it                      | You can open it up and fix it yourself                 |
| **Picture it as**             | A power tool in your hand                         | A machine on the factory floor                         |

The short version: **MCP for when you want to be in the cockpit, an API for autopilot.** If I want to sit and explore with it, MCP. If I want it to run on its own and just be right, a script.

A last word on the hype, because there's a lot of it. The real promise of MCP is that it's one standard that any tool and any model can speak, no custom wiring. The catch I kept hitting is that the quality is all over the map, and a flaky connection is more frustrating than a quick one-time script I can read and fix myself.
